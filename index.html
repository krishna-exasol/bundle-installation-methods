A “bundle” is a product made of several parts — a database, a server, a CLI — that must end up working together on a user’s machine. No single install method is best; there is a trade-off between shipping fast with low friction (script pipes, Docker) and trust, polish & discoverability (signed installers, OS package managers). This page catalogs the options so you can pick deliberately.

Why major players lead with curl | sh and irm | iex

The most common “Get started” command in modern developer tools is a single line that pipes a hosted install script into the shell. irm | iex is simply the PowerShell-native equivalent of curl | sh — Invoke-RestMethod downloads, Invoke-Expression runs — and PowerShell ships in Windows, so it needs nothing else.

# macOS / Linux
$ curl -fsSL https://example.com/install.sh | sh

# Windows PowerShell
PS> irm https://example.com/install.ps1 | iex

The benefits that make them converge on it

Why it wins
  • One line, zero prerequisites — no package manager, runtime, or registry account; a shell is always there.
  • Cross-platform parity — one .sh + one .ps1 cover every OS.
  • Adaptive — the script detects OS/arch/libc and fetches the right artifact.
  • Does the whole job — PATH, completions, services, config, multi-component startup, “what’s next”.
  • Ship & update instantly — no store review; you own the URL and the funnel.
  • Trivial in CI / Dockerfiles — drops into one RUN line.
The honest trade-offs
  • Trust — the user runs code they haven’t read (serve HTTPS from your domain; offer a read-before-run form).
  • No built-in integrity unless you add a checksum → the verified variant.
  • You own idempotency, upgrade, and uninstall.
  • Pointing at main lets behavior change under users — pin a release.

In one line: it is the lowest-friction, fastest-to-ship, fully-controllable path — which is why it wins for MVPs and stays as the universal fallback. Full write-up: Script pipe →

All 20 methods — pros & cons

How to choose

MVP / demo this week (multi-component) → a script pipe driving Docker Compose. Zero registry, cross-platform, full setup logic.
Hardening for early adopters → download → verify → run with a pinned release + SHA256.
Mainstream Windows → Winget (+ Scoop / Chocolatey).
Mainstream macOS / Linux → Homebrew (tap → core) and native deb/rpm.
Cloud / Kubernetes → Helm + IaC modules / marketplace images.

The “offer 2–3, not 10” rule: one zero-friction path for everyone, one native path per OS you care about, optionally one power-user path. More channels than that and they rot. See the decision guide and maturity roadmap.
